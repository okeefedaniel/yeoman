User Manual

Yeoman is the speaking-engagement and event-scheduling product in the DockLabs suite. It moves an invitation — for a hearing, keynote, panel, ribbon-cutting, site visit, fireside chat, or any other ask on the principal's time — through public intake, triage, decision, delegation, calendar push, and completion.

Yeoman is built around one core record (the Invitation) and the lifecycle a request travels through. It does not store bills, FOIA requests, or grants; those live in their owning peer products.

Two paths:

  1. Public intake form at /invite/. Anyone can submit. The submitter receives a status URL (/invite/status/<token>/) so they can check progress without an account.
  2. Programmatic intake at /api/v1/intake/invitation/ (bearer-token auth) for forwarding from another website or a legacy system.

Either way, the new invitation lands in the Received status, and an invitation_received notification fires to schedulers and administrators in your agency.

Full access. Triage, decide, delegate, schedule, complete, cancel, reopen. Manages the principal profile and reference addresses.
Scheduler
Operates the queue. Starts review, requests info, pushes to calendar, completes, and cancels. Cannot accept / decline / delegate — those are Administrator decisions.

Reads invitations delegated to them. Receives delegation notifications.
Viewer
Read-only across the dashboard, list, and calendar.

  • Assigned (coordinator) — the staff member managing logistics and submitter communication. Set by the Claim action.
  • Principal — the person speaking. Typically the Commissioner or agency head whose time is being scheduled.
  • Delegated — when the principal can't attend, an Administrator delegates to a deputy. The chain is captured in the delegation history.

From an Accepted, Tentative, or Delegated invitation, click Push to Calendar. Yeoman calls keel.calendar.push_event to create the event in the configured calendar (Outlook / Exchange / Google), stores the external event ID on the invitation (calendar_event_id), stamps the push timestamp, and moves the invitation to the Scheduled status.

Cancelling a Scheduled invitation calls back to cancel_event so the external event is removed.

  • Download an .ics file at /invitations/<uuid>/ical/ and forward it manually.
  • Send a calendar invite at /invitations/<uuid>/send-calendar/. Yeoman emails the invite (via keel.calendar) to one or more recipients and records who received it on the invitation for audit.

When an invitation is saved with a venue address but no coordinates, Yeoman geocodes the address automatically. The resulting latitude/longitude let the invitation appear as a marker on /map/, colored by status.

If geocoding fails, the invitation is still created — it just doesn't appear on the map until the address is corrected.

If the suite includes Beacon (the CRM), Administrators see an Add to Beacon control on each invitation. Clicking it pushes the submitter to Beacon as a contact, records the resulting Beacon contact ID on the invitation, and drops a provenance entry on the Beacon contact pointing back to this Yeoman invitation.

If Beacon isn't configured, the button is hidden — there are no silent no-ops. The control reappears once Beacon is reachable.

Helm's Today → Awaiting Me column queries each peer product through a per-user inbox endpoint. Yeoman exposes /api/v1/helm-feed/inbox/ and returns invitations where you are the gating dependency:

  • Coordinate — open invitations assigned to you.
  • Speak — open invitations where you are the principal.
  • Cover for — open invitations delegated to you.

Open your sidebar profile dropdown and click Notification Preferences (or visit /notifications/preferences/).

For each event type (invitation_received, invitation_assigned, invitation_delegated, etc.), toggle the in-app and email channels independently.

Administrators visit Settings → Principal (/settings/principal/) to set the display name, title, contact info, and notes for the principal. Reference addresses (Home, Office, Capitol, etc.) are managed on the same page; one can be marked the default. Yeoman uses these to compute driving distance from incoming invitation venues.

There is one principal profile per agency.

The intake form returns a status URL of the form /invite/status/<status_token>/. Anyone holding that URL can see the current status, the assigned coordinator's name (when available), and the event date once confirmed. They cannot see internal notes, attachments other than ones they uploaded themselves, or any information about other invitations.

The token is per-invitation, so sharing it doesn't expose the rest of the queue.
